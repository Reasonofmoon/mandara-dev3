# Hydration 불일치 SSR 호환 수정 적용

진단된 Hydration 불일치 원인(브라우저 API 접근, 시간/랜덤값, 조건부 렌더, 클라이언트 전용 라이브러리)에 대해 SSR 호환 수정안을 트레이드오프와 함께 제시하고, SEO/초기 렌더 영향을 최소화하며 안전하게 구현하도록 지시하는 프롬프트입니다.

## 용도

F-03 플로우의 "6단계: 수정안 선택"을 근거로, 서버 HTML과 클라이언트 렌더 결과를 일치시킵니다. 지연 렌더, 동적 임포트, suppressHydrationWarning 등 후보를 SEO·깜빡임 트레이드오프와 함께 비교합니다.

## 적용 시점

- "Hydration failed because the initial UI does not match" 원인이 특정됐을 때
- localStorage/window 등 브라우저 API가 렌더 경로에서 발견됐을 때
- 서드파티 라이브러리를 SSR에서 제외할지 결정할 때

## 필수 입력

- 진단된 원인 (예: 렌더 중 new Date() 사용 / localStorage 접근)
- 문제 컴포넌트와 불일치가 발생하는 DOM 노드
- SEO 요구사항 (해당 콘텐츠가 검색 인덱싱 대상인지)

## 프롬프트 템플릿

```
다음 Hydration 불일치 원인에 대한 수정안을 적용하라.

[진단된 원인]
- 컴포넌트: {컴포넌트명}
- 근본 원인: {예: 렌더 중 Math.random() / localStorage.getItem() 직접 호출}
- SEO 대상 여부: {예/아니오}

[요구사항]
1. 아래 수정안 후보를 트레이드오프와 함께 비교하라:
   - 수정안 A: useEffect + isMounted 지연 렌더 (간단, 초기 깜빡임/CLS 위험)
   - 수정안 B: next/dynamic ssr:false 동적 임포트 (완전 격리, 초기 HTML에서 콘텐츠 누락 → SEO 손실)
   - 수정안 C: 서버/클라이언트 동일 고정값 사용 (불일치 원천 제거, 동적성 상실)
   - 수정안 D: suppressHydrationWarning (부득이한 경우만, 경고 은폐 위험)
2. 최소 침습 원칙: 불일치 노드만 격리하고, 페이지 전체를 클라이언트 렌더로 돌리지 마라.
3. 사이드 이펙트를 점검하라:
   - 초기 렌더 깜빡임(CLS) 및 로딩 플레이스홀더 필요 여부
   - SSR 제외 시 SEO/초기 콘텐츠 영향
4. 롤백 가능성 확보: SSR 전략 변경을 컴포넌트 단위로 되돌릴 수 있게 하라.

[출력]
- 선택한 수정안과 근거 (SEO 영향 명시)
- 수정 전/후 코드 diff
- 사이드 이펙트 점검 결과
- 롤백 절차
```

## 출력 계약

- 선택한 수정안과 근거 (SEO/깜빡임 트레이드오프 명시)
- 수정 전/후 코드 diff
- 사이드 이펙트 점검표: CLS, 플레이스홀더, SSR 콘텐츠 노출
- 롤백 절차: 컴포넌트 단위 SSR 전략 복귀 방법

- [ ] 불일치 노드만 격리하고 페이지 전체를 CSR로 돌리지 않았는가?
