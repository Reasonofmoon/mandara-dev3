# N+1 쿼리 배치 최적화 수정 적용

진단된 N+1 원인(루프 내 쿼리, 관계 개별 조회, include 누락)에 대해 쿼리 배치 수정안을 트레이드오프와 함께 제시하고, 결과 정확성을 유지하며 안전하게 구현하도록 지시하는 프롬프트입니다.

## 용도

F-09 플로우의 "6단계: 수정안 선택"을 근거로, 반복 쿼리를 배치/조인으로 축소합니다. Prisma include/select, groupBy 배치, raw SQL, DataLoader, 캐싱 등 후보를 성능·메모리·정확성 트레이드오프와 함께 비교합니다.

## 적용 시점

- 목록 조회에서 쿼리 수가 항목 수에 비례해 증가하는 원인이 특정됐을 때
- 관계 데이터를 루프에서 개별 조회하고 있을 때
- GraphQL 리졸버에서 필드별 쿼리가 발생할 때

## 필수 입력

- 진단된 원인 (예: users 루프 내 posts 개별 조회)
- 관계 구조(1:N, M:N)와 조회에 필요한 필드
- 데이터 규모와 목표 쿼리 수/응답 시간

## 프롬프트 템플릿

```
다음 N+1 쿼리 원인에 대한 수정안을 적용하라.

[진단된 원인]
- 위치: {예: getUsersWithPosts 함수}
- 근본 원인: {예: user 루프마다 post.findMany 호출로 N+1}
- 관계/필드: {예: User 1:N Post, 필요한 필드 id/title}

[요구사항]
1. 아래 수정안 후보를 트레이드오프와 함께 비교하라:
   - 수정안 A: Prisma include로 관계 일괄 로드 (간단, 과다 로드 시 페이로드 증가)
   - 수정안 B: select로 필요한 필드만 로드 (전송량 절감, 필드 누락 위험)
   - 수정안 C: groupBy/aggregate로 카운트만 배치 (집계 전용, 상세 데이터 별도)
   - 수정안 D: raw SQL 조인 + JSON 집계 (최고 성능, 타입 안전성 상실·유지보수 부담)
   - 수정안 E: DataLoader 배치 (GraphQL 리졸버 적합, 요청 스코프 관리 필요)
   - 수정안 F: Redis 캐싱 (반복 조회 완화, 캐시 무효화 정책 필요)
2. 최소 침습 원칙: 결과 형태(반환 스키마)를 유지한 채 쿼리 실행 방식만 배치로 바꿔라.
3. 사이드 이펙트를 점검하라:
   - include/select 변경으로 반환 데이터 형태가 바뀌어 호출부 깨짐 여부
   - 대량 include로 인한 메모리/페이로드 증가
   - 캐싱 도입 시 정합성(무효화 시점)과 stale 데이터
4. 롤백 가능성 확보: 쿼리 함수 단위로 교체해 되돌릴 수 있게 하라.

[출력]
- 선택한 수정안과 근거
- 쿼리 함수 수정 전/후 코드 diff
- 사이드 이펙트 점검 결과 (쿼리 수 before/after 포함)
- 롤백 절차
```

## 출력 계약

- 선택한 배치 수정안과 근거 (관계 로딩 전략 포함)
- 쿼리 함수 수정 전/후 코드 diff
- 사이드 이펙트 점검표: 반환 형태 호환성, 페이로드/메모리, 캐시 정합성
- 쿼리 수 측정: 수정 전 N+1 → 수정 후 실제 쿼리 수
- 롤백 절차: 쿼리 함수 단위 복귀 방법

## 셀프 체크리스트

- [ ] 루프 내 개별 쿼리를 제거하고 배치/조인으로 전환했는가?
- [ ] 반환 데이터 형태를 유지해 호출부가 깨지지 않는가?
- [ ] 수정 후 쿼리 수가 항목 수와 무관하게 상수 수준인가?
- [ ] 과다 include로 불필요한 데이터를 로드하지 않는가?
- [ ] 캐싱 도입 시 무효화 정책으로 stale 데이터를 방지하는가?
- [ ] 쿼리 로깅으로 before/after 쿼리 수를 검증했는가?
