# JWT 토큰 관리 수정 적용

진단된 JWT 문제(시크릿 오류, 만료 검증 누락, 불안전한 저장 위치, 리프레시 미구현)에 대해 토큰 관리 수정안을 트레이드오프와 함께 제시하고, 보안(XSS/CSRF)을 유지하며 안전하게 구현하도록 지시하는 프롬프트입니다.

## 용도

F-07 플로우의 "6단계: 수정안 선택"을 근거로, 액세스/리프레시 토큰 쌍 발급·검증·갱신을 정비합니다. 저장 위치(HttpOnly 쿠키 vs 메모리 vs localStorage)와 갱신 로직을 보안 트레이드오프와 함께 비교합니다.

## 적용 시점

- "Invalid signature"/만료 미검증 원인이 특정됐을 때
- 토큰이 localStorage에 저장되어 XSS에 노출될 때
- 리프레시 토큰 흐름을 도입/정비할지 결정할 때

## 필수 입력

- 진단된 원인 (예: 액세스 토큰 만료가 너무 길고 리프레시 없음)
- 현재 토큰 저장 위치와 전송 방식
- 보안 요구사항 (XSS/CSRF 위협 모델, HTTPS 여부)

## 프롬프트 템플릿

```
다음 JWT 토큰 문제 원인에 대한 수정안을 적용하라.

[진단된 원인]
- 대상: {예: 로그인/토큰 갱신 엔드포인트}
- 근본 원인: {예: localStorage 저장 + 리프레시 미구현}
- 저장/전송: {현재 방식}

[요구사항]
1. 아래 수정안 후보를 트레이드오프와 함께 비교하라:
   - 수정안 A: 액세스(단명, 15m) + 리프레시(장수, 7d) 토큰 쌍 발급, 서로 다른 시크릿 사용
   - 수정안 B: HttpOnly + Secure + SameSite 쿠키 저장 (XSS 방어 강함, CSRF 대비 필요)
   - 수정안 C: 액세스는 메모리 변수 + 리프레시는 서버 쿠키 (XSS 표면 축소, 새로고침 시 재발급)
   - 수정안 D: 리프레시 토큰 회전 + 블랙리스트 (재사용 탐지, 상태 저장 부담)
2. 최소 침습 원칙: localStorage 저장을 제거하고, 검증 옵션(issuer/audience/expiresIn)을 명시하라.
3. 사이드 이펙트를 점검하라:
   - HttpOnly 쿠키 도입 시 클라이언트 fetch에 credentials:'include' 필요
   - SameSite 설정과 교차 사이트 요청 영향
   - 리프레시 회전 시 동시 요청 레이스와 로그아웃 처리
4. 롤백 가능성 확보: 토큰 발급/저장 방식을 authService 단일 지점으로 격리하라.

[출력]
- 선택한 수정안과 근거 (저장 위치 결정 포함)
- authService/미들웨어/쿠키 설정 수정 전/후 코드 diff
- 보안 사이드 이펙트 점검 결과
- 롤백 절차
```

## 출력 계약

- 선택한 토큰 관리 수정안과 근거 (저장 위치·수명 정책 포함)
- 토큰 발급/검증/갱신 수정 전/후 코드 diff
- 보안 점검표: XSS/CSRF, SameSite, 시크릿 분리, 만료 검증
- 롤백 절차: authService 단일 지점 복귀 방법

## 셀프 체크리스트

- [ ] 액세스/리프레시 토큰을 분리하고 서로 다른 시크릿을 쓰는가?
- [ ] 토큰을 localStorage가 아닌 HttpOnly 쿠키 또는 메모리에 저장하는가?
- [ ] 검증 시 만료·issuer·audience를 함께 확인하는가?
- [ ] SameSite/Secure 설정으로 CSRF·전송 위협을 완화했는가?
- [ ] JWT_SECRET이 충분히 길고(32자+) 환경변수로 관리되는가?
- [ ] 토큰 발급/저장 로직이 단일 지점으로 격리되어 롤백 가능한가?
