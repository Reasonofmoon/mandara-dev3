# 인증/인가 가드 수정 적용

진단된 인증/인가 문제(401과 403 미구분, 토큰 검증 누락, 역할 검사 부재)에 대해 서버 가드·클라이언트 처리 수정안을 트레이드오프와 함께 제시하고, 보안 경계를 유지하며 안전하게 구현하도록 지시하는 프롬프트입니다.

## 용도

F-06 플로우의 "6단계: 수정안 선택"을 근거로, 401(인증)과 403(인가)을 정확히 분기하고 RBAC를 적용합니다. 서버 미들웨어/가드, 클라이언트 에러 처리, 라우트 보호 등 후보를 트레이드오프와 함께 비교합니다.

## 적용 시점

- 401과 403이 같은 처리로 뭉개지는 원인이 특정됐을 때
- 토큰은 유효하나 역할 검사가 없어 권한 상승이 가능할 때
- 클라이언트에서 만료(401)와 권한 부족(403)을 구분해 처리해야 할 때

## 필수 입력

- 진단된 원인 (예: authorize 미들웨어 부재로 role 검사 없음)
- 보호 대상 엔드포인트/라우트와 요구 역할
- 서버 프레임워크(Express/NestJS)와 클라이언트 라우팅 방식

## 프롬프트 템플릿

```
다음 인증/인가 오류 원인에 대한 수정안을 적용하라.

[진단된 원인]
- 대상: {예: DELETE /api/admin, /admin 라우트}
- 근본 원인: {예: 토큰 검증만 있고 역할 검사 없음 / 401·403 미구분}
- 요구 역할: {예: admin}

[요구사항]
1. 아래 수정안 후보를 트레이드오프와 함께 비교하라:
   - 수정안 A: 서버 authenticateToken + authorize(role) 미들웨어 분리 (명확한 401/403 분기)
   - 수정안 B: NestJS JwtAuthGuard + RolesGuard + @Roles 데코레이터 (선언적, 프레임워크 결합)
   - 수정안 C: 클라이언트 fetchWithAuth로 401→재로그인 / 403→권한 안내 분기
   - 수정안 D: React ProtectedRoute로 라우트 레벨 보호 (UX 개선, 서버 검증 병행 필수)
2. 최소 침습 원칙: 인증(401)과 인가(403)를 분리된 단계로 처리하고, 서버 검증을 최종 방어선으로 유지하라.
3. 사이드 이펙트를 점검하라:
   - 토큰 만료 시 401 반환과 클라이언트 재로그인 흐름 일관성
   - 403 응답에 required/current role 노출 시 정보 누출 여부
   - 클라이언트 라우트 보호만으로 서버 보호를 대체하지 않았는지
4. 롤백 가능성 확보: 가드를 엔드포인트 단위로 붙여 되돌릴 수 있게 하라.

[출력]
- 선택한 수정안과 근거
- 미들웨어/가드 수정 전/후 코드 diff
- 사이드 이펙트 점검 결과 (정보 누출, 서버-클라이언트 이중 방어)
- 롤백 절차
```

## 출력 계약

- 선택한 가드 수정안과 근거 (401/403 분기 정책 포함)
- 서버 미들웨어/가드 및 클라이언트 처리 수정 전/후 코드 diff
- 사이드 이펙트 점검표: 만료 흐름, 역할 정보 노출, 이중 방어
- 롤백 절차: 엔드포인트 단위 가드 제거 방법

## 셀프 체크리스트

- [ ] 인증 실패는 401, 권한 부족은 403으로 정확히 분기하는가?
- [ ] 토큰 만료가 401로 반환되고 클라이언트가 재로그인/갱신으로 이어지는가?
- [ ] 역할 검사(RBAC)가 서버측에서 강제되는가?
- [ ] 클라이언트 라우트 보호가 서버 검증을 대체하지 않는가?
- [ ] 403 응답이 불필요한 내부 역할 정보를 노출하지 않는가?
- [ ] 가드를 엔드포인트 단위로 롤백할 수 있는가?
