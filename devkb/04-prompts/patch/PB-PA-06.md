# API 멱등성 보장 수정 적용

진단된 멱등성 문제(Idempotency-Key 미구현, 중복 검사 부재, 재시도 부작용)에 대해 멱등성 보장 수정안을 트레이드오프와 함께 제시하고, 중복 처리(중복 결제/주문)를 막으며 안전하게 구현하도록 지시하는 프롬프트입니다.

## 용도

F-05 플로우의 "6단계: 수정안 선택"을 근거로, 동일 요청 재실행 시 부작용이 없도록 만듭니다. Idempotency-Key, DB 유니크 제약, 클라이언트 재시도 등 후보를 저장소·정합성 트레이드오프와 함께 비교합니다.

## 적용 시점

- 네트워크 재시도로 중복 리소스가 생성되는 원인이 특정됐을 때
- 결제/주문 등 부작용 있는 POST에 멱등 키가 없을 때
- Idempotency-Key 저장을 Redis/DB 중 어디에 둘지 결정할 때

## 필수 입력

- 진단된 원인 (예: POST /orders에 중복 검사 없음)
- 대상 엔드포인트와 부작용의 위험도 (결제/주문/기타)
- 사용 가능한 저장소 (Redis, DB 유니크 제약) 및 TTL 요구

## 프롬프트 템플릿

```
다음 멱등성 문제 원인에 대한 수정안을 적용하라.

[진단된 원인]
- 엔드포인트: {예: POST /api/orders}
- 근본 원인: {예: Idempotency-Key 미검증으로 재시도 시 중복 생성}
- 부작용 위험: {결제/주문/기타}

[요구사항]
1. 아래 수정안 후보를 트레이드오프와 함께 비교하라:
   - 수정안 A: Idempotency-Key + Redis 캐시 (빠름, 캐시 만료/유실 시 재검증 필요)
   - 수정안 B: DB 유니크 제약(idempotency_key UNIQUE) (강한 정합성, 충돌 에러 처리 필요)
   - 수정안 C: 클라이언트 지수 백오프 재시도 + 키 재사용 (서버 부담 완화)
   - 수정안 D: A+B 병행 (캐시 히트 + DB 최종 보증, 구현 복잡도 상승)
2. 최소 침습 원칙: 부작용 있는 메서드(POST/PUT/DELETE)에만 멱등 미들웨어를 적용하라.
3. 사이드 이펙트를 점검하라:
   - 저장된 응답과 재요청 응답의 상태코드/헤더(Idempotency-Replay) 일관성
   - 키 TTL 만료 후 재요청 시 이중 처리 가능성
   - 동시 요청(race) 시 유니크 제약 충돌 처리
4. 롤백 가능성 확보: 멱등 미들웨어를 라우트 단위로 분리해 되돌릴 수 있게 하라.

[출력]
- 선택한 수정안과 근거
- 미들웨어/스키마 수정 전/후 코드 diff
- 사이드 이펙트 점검 결과 (TTL, race, 응답 일관성)
- 롤백 절차
```

## 출력 계약

- 선택한 멱등성 보장 수정안과 근거 (저장소 선택 포함)
- 미들웨어/DB 제약 수정 전/후 코드 diff
- 사이드 이펙트 점검표: TTL 만료, 동시성 충돌, replay 응답 일관성
- 롤백 절차: 라우트 단위 멱등 로직 제거 방법

## 셀프 체크리스트

- [ ] 부작용 있는 메서드에만 멱등성 검사를 적용했는가?
- [ ] 동일 키 재요청이 같은 리소스 ID와 응답을 반환하는가?
- [ ] Idempotency-Replay 헤더로 재실행을 식별할 수 있는가?
- [ ] DB 유니크 제약으로 동시성 충돌을 최종 방어하는가?
- [ ] 키 TTL 만료 후 이중 처리 위험을 검토했는가?
- [ ] 멱등 미들웨어를 라우트 단위로 롤백할 수 있는가?
