# React 무한 리렌더링 수정 적용

진단된 무한 리렌더링 원인(useEffect 의존성, 객체 참조 불안정, 프롭 전달 등)에 대해 수정안 옵션을 트레이드오프와 함께 제시하고, 롤백 가능한 최소 변경으로 구현하도록 지시하는 프롬프트입니다.

## 용도

F-01 플로우의 "6단계: 수정안 선택"을 근거로, 재렌더 루프를 끊는 구체적 수정을 안전하게 반영합니다. 여러 후보 수정을 나열하고 각각의 부작용을 밝힌 뒤, 가장 적은 코드 변경으로 문제를 해결하도록 유도합니다.

## 적용 시점

- 렌더 카운트 급증 원인이 특정 컴포넌트/훅으로 좁혀졌을 때
- "Maximum update depth exceeded" 원인 후보가 정렬되었을 때
- useEffect/useMemo/useCallback 중 어떤 것을 손댈지 결정해야 할 때

## 필수 입력

- 진단된 근본 원인 (예: useEffect 의존성 누락 / 상위 객체 참조 변화)
- 문제 컴포넌트의 렌더 트리거 상태 및 프롭
- 현재 렌더 카운트와 목표 렌더 카운트

## 프롬프트 템플릿

```
다음 무한 리렌더링 원인에 대한 수정안을 적용하라.

[진단된 원인]
- 컴포넌트: {컴포넌트명}
- 근본 원인: {예: useEffect 의존성 배열 누락으로 setState 루프}
- 관련 상태/프롭: {상태, 프롭 목록}

[요구사항]
1. 아래 수정안 후보를 트레이드오프와 함께 비교하라:
   - 수정안 A: useEffect 의존성 배열 추가 (변경 최소, 의존성 오탐 위험)
   - 수정안 B: 의존성 객체를 useMemo로 안정화 (참조 고정, 메모리 소폭 증가)
   - 수정안 C: 콜백을 useCallback으로 안정화 (자식 리렌더 방지, 의존성 관리 필요)
   - 수정안 D: setState를 함수형 업데이트로 전환 (이전 상태 의존 제거)
2. 최소 침습 원칙: 렌더 루프를 끊는 데 필요한 최소 변경만 적용하고, 무관한 리팩터링은 하지 마라.
3. 각 수정에 대해 사이드 이펙트를 점검하라:
   - 의존성 추가/제거로 인한 effect 실행 시점 변화
   - 메모이제이션으로 stale closure 발생 여부
4. 롤백 가능성 확보: 변경을 단일 커밋으로 격리하고, 되돌릴 경우의 동작을 명시하라.

[출력]
- 선택한 수정안과 근거
- 수정 전/후 코드 diff
- 사이드 이펙트 점검 결과
- 롤백 절차
```

## 출력 계약

- 선택한 수정안 1개(또는 조합)와 선택 근거
- 수정 전/후 코드 diff (변경 라인만)
- 사이드 이펙트 점검표: effect 실행 타이밍, stale closure, 자식 리렌더 영향
- 롤백 절차: 되돌릴 커밋과 예상 복귀 동작

## 셀프 체크리스트

- [ ] 최소 변경으로 렌더 루프를 끊었는가? (무관한 리팩터링 없음)
- [ ] 의존성 배열 변경이 새로운 stale closure를 만들지 않는가?
- [ ] useMemo/useCallback 도입 시 의존성이 정확한가?
- [ ] 수정 후 렌더 카운트가 목표 이하로 안정되는가?
- [ ] 변경을 단일 커밋으로 격리해 롤백 가능한가?
- [ ] react-hooks/exhaustive-deps 린트 경고가 남지 않는가?
