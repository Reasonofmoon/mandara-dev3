# 느린 쿼리 재현 절차 수립

특정 쿼리가 임계 시간을 넘겨 느려지는 문제를, EXPLAIN ANALYZE로 실행 계획과 실측 시간을 고정 캡처해 재현하는 프롬프트입니다.

## 용도

"어떤 페이지가 느리다"를, 원인 쿼리 하나로 좁혀 동일 파라미터·동일 데이터 조건에서 실행 계획(Seq Scan 등)과 실행 시간을 반복 재현 가능하게 만듭니다.

## 적용 시점

- 쿼리 실행 시간이 임계치(예: 2초)를 넘길 때
- Seq Scan/전체 테이블 스캔이 의심될 때
- 동시 접속 증가 시 특정 쿼리만 느려질 때
- 인덱스 추가 전후를 비교할 기준선이 필요할 때

## 필수 입력

- 느린 쿼리 원문과 바인딩 파라미터
- 대상 테이블 행 수와 통계 최신 여부
- DBMS 종류(PostgreSQL/MySQL)
- 현재 관측된 실행 시간

## 프롬프트 템플릿

아래 지시를 AI 도구나 팀원에게 그대로 전달하세요.

```
다음 느린 쿼리를 EXPLAIN ANALYZE로 조건을 고정해 재현하고, 실행 계획과 시간을 기록해줘.

[대상]
- 쿼리: (원문 붙여넣기)
- 파라미터: (예: author_id = 1)
- DBMS: (PostgreSQL / MySQL)

[재현 절차 지시]
1. 재현 조건부터 고정:
   - 통계 상태를 확정: PostgreSQL은 ANALYZE <table>; 실행 후/전을 구분 기록.
   - 캐시 영향 배제를 위해 동일 쿼리를 2~3회 실행하고 2회차 시간을 기준으로 삼음.
2. 실행 계획과 실측 시간을 캡처:
   -- PostgreSQL
   EXPLAIN (ANALYZE, BUFFERS) <쿼리>;
   -- MySQL
   EXPLAIN ANALYZE <쿼리>;   -- 또는 EXPLAIN FORMAT=JSON <쿼리>;
   → Seq Scan / Index Scan, rows, Execution Time 을 기록.
3. 파라미터를 고정한 채 데이터 건수만 바꿔 재현:
   - 대상 테이블 행 수(현재 / 10배)에서 Execution Time 변화를 측정.
4. 스캔 유형을 분류:
   (a) WHERE 컬럼 인덱스 부재로 Seq Scan
   (b) 큰 테이블 조인으로 Hash Join 비용 큼
   (c) 전체 스캔(필터 부재)
   (d) 통계 오래됨으로 잘못된 계획.

[재현 조건 고정]
- 재현에 쓴 정확한 쿼리와 파라미터를 명시.
- EXPLAIN ANALYZE 출력 전문을 첨부(계획 노드+시간).
- 동일 조건 재실행 시 시간이 재현되는지(2회차 기준) 확인.
```

## 출력 계약

- EXPLAIN ANALYZE 출력 전문(계획 노드 + Execution Time)
- 스캔 유형(a~d 중) 확정 및 근거 노드
- 데이터 건수 변화에 따른 실행 시간 표
- 결정론적 재현 쿼리(파라미터 고정, 복붙용)
