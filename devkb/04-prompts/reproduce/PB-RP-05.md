# API 멱등성 결여 재현 절차 수립

동일 요청을 여러 번 보내거나 재시도할 때 주문/결제가 중복 생성되는 멱등성 문제를, 같은 Idempotency-Key로 반복 호출해 재현하는 프롬프트입니다.

## 용도

"재시도했더니 중복 처리됨" 같은 산발적 사고를, 동일 페이로드+동일 키의 반복 호출로 결정론적으로 재현하고 생성 개수를 세어 문제를 수치화합니다.

## 적용 시점

- 네트워크 오류 후 재시도에서 리소스가 중복 생성될 때
- POST 엔드포인트가 매번 새 레코드를 만들 때
- Idempotency-Key 처리 여부가 불명확할 때
- 재시도 로직 도입 전 안전성을 확인해야 할 때

## 필수 입력

- 대상 엔드포인트와 요청 페이로드
- Idempotency-Key 헤더 지원 여부
- 중복 판정 기준(같은 주문 ID여야 함 등)
- 현재 관측된 중복 증상(결제 N회 등)

## 프롬프트 템플릿

아래 지시를 AI 도구나 팀원에게 그대로 전달하세요.

```
다음 멱등성 문제를 동일 키 반복 호출로 재현하고, 생성 개수를 기록해줘.

[증상]
- 엔드포인트: (예: POST /api/orders)
- 페이로드: (JSON)
- 증상: (예: 재시도 시 주문 2건 생성)

[재현 절차 지시]
1. 동일 Idempotency-Key로 같은 요청을 3회 연속 호출:
   for i in 1 2 3; do
     curl -s -X POST '<endpoint>' \
       -H 'Content-Type: application/json' \
       -H 'Idempotency-Key: repro-key-001' \
       -d '<payload>' ; echo;
   done
   → 반환된 리소스 ID 3개가 모두 동일한지 확인.
2. 키 없이 같은 요청 3회 호출해 대조군 확보:
   → 키가 없으면 중복 생성되는지 확인(현재 동작 baseline).
3. 재시도 시나리오 재현: 응답을 강제로 무시하고 즉시 재호출해
   첫 요청이 성공했는데도 두 번째가 새 리소스를 만드는지 확인.
4. DB에서 실제 생성된 행 수를 직접 조회해 검증:
   SELECT id, idempotency_key, created_at FROM orders
   WHERE idempotency_key = 'repro-key-001' OR <payload 조건>;

[재현 조건 고정]
- 재현에 쓴 정확한 키/페이로드/호출 횟수를 명시.
- "기대 1건 vs 실제 N건" 형식으로 결과를 수치화.
- 키 있음/없음 두 조건 각각의 생성 개수를 표로 기록.
```

## 출력 계약

- 동일 키 3회 호출 결과의 리소스 ID 목록
- 키 있음/없음 대조군의 생성 개수 표(기대 vs 실제)
- DB에서 확인한 실제 행 수
- 결정론적 재현 명령(복붙용 loop)
