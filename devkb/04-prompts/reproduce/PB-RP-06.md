# 인증/인가 오류 분기 재현 절차 수립

토큰 누락·만료·권한 부족이 뒤섞여 나타나는 401/403 오류를, 각 조건을 개별 curl 요청으로 분리해 상태 코드가 정확히 갈리는지 재현하는 프롬프트입니다.

## 용도

"접근이 안 됨"이 인증 문제(401)인지 권한 문제(403)인지를 조건별로 고정 재현해, 서버가 상태 코드를 올바르게 분기하는지 확인합니다.

## 적용 시점

- 401과 403이 같은 처리로 뭉뚱그려질 때
- 토큰 만료가 401로 내려오는지 확인해야 할 때
- 일반 사용자 토큰으로 관리자 리소스 접근이 403인지 검증할 때
- 역할 기반 접근(RBAC) 경계를 재현해야 할 때

## 필수 입력

- 보호된 엔드포인트와 필요한 역할
- 유효/만료/잘못된 토큰 각 샘플
- 일반 사용자 토큰과 관리자 토큰
- 현재 관측된 상태 코드와 응답 바디

## 프롬프트 템플릿

아래 지시를 AI 도구나 팀원에게 그대로 전달하세요.

```
다음 인증/인가 문제를 조건별로 분리 재현하고, 각 케이스의 상태 코드를 표로 정리해줘.

[증상]
- 엔드포인트: (예: DELETE /api/admin)
- 필요한 역할: (예: admin)
- 현재 증상: (예: 401/403 구분 없이 실패)

[재현 절차 지시 - 5개 조건을 개별 요청으로]
1. 토큰 없음 → 401 기대:
   curl -i -X <메서드> '<endpoint>'
2. 잘못된 토큰 → 401 기대:
   curl -i -X <메서드> '<endpoint>' -H 'Authorization: Bearer invalid.token.here'
3. 만료된 토큰 → 401(TokenExpired) 기대:
   curl -i -X <메서드> '<endpoint>' -H 'Authorization: Bearer <만료 토큰>'
4. 유효하나 권한 부족(일반 사용자) → 403 기대:
   curl -i -X <메서드> '<endpoint>' -H 'Authorization: Bearer <user 토큰>'
5. 올바른 역할(관리자) → 200 기대:
   curl -i -X <메서드> '<endpoint>' -H 'Authorization: Bearer <admin 토큰>'

[분기 검증]
- 각 요청의 상태 코드와 응답 바디의 error/code 필드를 캡처.
- 401이어야 할 케이스가 403으로, 또는 그 반대로 새어나오는지 확인.

[재현 조건 고정]
- 각 조건에 쓴 토큰의 출처/만료시각을 명시.
- 5개 조건 × 상태코드 매트릭스로 재현 결과를 표로 기록.
- 반복 실행 시 상태 코드가 동일하게 나오는지(결정론) 확인.
```

## 출력 계약

- 5개 조건 각각의 curl 명령과 실제 상태 코드
- "조건 → 기대 코드 → 실제 코드" 매트릭스
- 401/403이 잘못 교차하는 케이스 지목
- 결정론적 재현 명령 세트(복붙용)

- [ ] 각 조건의 기대 코드와 실제 코드를 대조했는가?
