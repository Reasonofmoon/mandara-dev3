# N+1 쿼리 재현 절차 수립

관계 데이터를 개별 조회하며 쿼리 수가 N+1로 폭증하는 문제를, Prisma 쿼리 로그를 켜고 실제 발생 쿼리 수를 세어 재현하는 프롬프트입니다.

## 용도

"페이지가 느리다"를, 특정 조회 경로에서 발생하는 쿼리 개수를 데이터 건수에 따라 측정해 N+1 패턴임을 수치로 확정합니다.

## 적용 시점

- 작은 데이터 조회에도 쿼리 수가 수백 개일 때
- 루프 안에서 관계 데이터를 개별 조회하는 코드가 의심될 때
- include/select 누락으로 지연 로딩이 발생할 때
- 데이터가 늘수록 응답이 급격히 느려질 때

## 필수 입력

- 대상 조회 함수/엔드포인트
- 관련 모델과 관계(1:N, 중첩 관계)
- 시드 데이터 건수(사용자 N명, 게시글 M개 등)
- 관측된 지연 시간

## 프롬프트 템플릿

아래 지시를 AI 도구나 팀원에게 그대로 전달하세요.

```
다음 조회의 N+1 여부를 쿼리 카운트 측정으로 재현하고, 데이터 건수와 쿼리 수의 관계를 기록해줘.

[대상]
- 조회 경로/함수: (예: getUsersWithPosts)
- 관계: (예: User 1:N Post, Post 1:N Comment)

[재현 절차 지시]
1. 쿼리 로깅을 켜고 카운터를 붙여 실제 발생 쿼리 수를 센다:
   const prisma = new PrismaClient({ log: [{emit:'event',level:'query'}] });
   let q = 0; prisma.$on('query', () => q++);
   await 대상조회함수();
   console.log('queries =', q);
2. 데이터 건수를 바꿔가며 쿼리 수의 증가를 측정:
   - 사용자 1명 / 10명 / 50명 시드에서 각각 q 값을 기록.
   - q 가 (1 + N) 또는 (1 + N + N*M) 형태로 선형 증가하면 N+1 확정.
3. 문제 경로를 최소 재현 스크립트로 축소:
   - 관계 include 없이 findMany 후 루프에서 개별 조회하는 형태로 고정.
4. 대조군: 동일 조회를 include 한 번으로 바꿔 q 가 상수(≈1)로 떨어지는지 확인.

[재현 조건 고정]
- 각 데이터 건수별 (건수 → 쿼리 수)를 표로 기록.
- 발생한 쿼리 로그 일부(반복되는 SELECT)를 첨부.
- 시드 스크립트와 조회 코드를 그대로 첨부해 재현 가능하게.
```

## 출력 계약

- 데이터 건수별 쿼리 수 표(1명/10명/50명 등)
- N+1 증가 공식(1+N 또는 1+N+N*M) 확정
- 반복되는 SELECT 쿼리 로그 발췌
- 최소 재현 스크립트(시드+조회)
