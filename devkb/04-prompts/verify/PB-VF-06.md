# 인증/인가(401·403) 분기 해결 검증

인증 미들웨어와 역할 기반 접근 제어(RBAC)를 수정한 뒤, 인증 실패는 401·권한 부족은 403으로 정확히 분기되고 역할별 접근 권한이 의도대로 적용되며 토큰 만료 시 적절히 처리되는지 확인하는 검증 프롬프트입니다.

## 용도

F-06 플로우의 6단계(수정) 이후, 상태 코드 분기의 정확성과 RBAC 권한 매트릭스를 회귀 테스트로 보증하고 클라이언트 리다이렉트 동작을 점검합니다.

## 적용 시점

- 401/403을 구분 처리하도록 미들웨어/가드를 수정한 직후
- 역할별 접근 정책(authorize, RolesGuard)을 변경한 직후
- 보호 라우트/리다이렉트 로직을 수정한 뒤

## 필수 입력

- 보호 엔드포인트와 각 엔드포인트에 요구되는 역할
- 테스트용 토큰 (미인증/일반 user/admin/만료)
- 401 vs 403 기대 응답 포맷

## 프롬프트 템플릿

아래 인증/인가 수정에 대해 상태 코드 분기와 RBAC를 검증하고 회귀가 없는지 확인해줘.

**엔드포인트:** [/api/protected(인증), /api/admin(admin 전용)]
**토큰 종류:** [없음, user, admin, 만료]

1. **성공 기준 정의**
   - 토큰 없음/무효/만료 → 401
   - 인증됐으나 권한 부족 → 403 (401 아님)
   - 올바른 역할 → 200
   - 403 응답에 required/current 역할 정보 포함, 401은 리소스 존재 노출 금지

2. **인증·인가 회귀 매트릭스**
   | 토큰 | 엔드포인트 | 기대 코드 | 결과 |
   |------|-----------|----------|------|
   | 없음 | /api/protected | 401 | |
   | 무효 | /api/protected | 401 | |
   | 만료 | /api/protected | 401 (TOKEN_EXPIRED) | |
   | user | /api/protected | 200 | |
   | user | /api/admin | 403 | |
   | admin | /api/admin | 200 | |

3. **자동화 테스트**
   ```javascript
   describe('인증/인가 분기', () => {
     it('토큰 없으면 401', async () =>
       expect((await fetch('/api/protected')).status).toBe(401));
     it('권한 부족이면 403', async () =>
       expect((await fetch('/api/admin',
         { headers: { Authorization: `Bearer ${userToken}` } })).status).toBe(403));
     it('올바른 역할이면 200', async () =>
       expect((await fetch('/api/admin',
         { headers: { Authorization: `Bearer ${adminToken}` } })).status).toBe(200));
   });
   ```

4. **클라이언트 처리 확인**
   - 401 → 토큰 제거 후 /login 리다이렉트
   - 403 → /forbidden 안내 (재로그인 유도 아님)
   - ProtectedRoute의 requiredRole 분기가 동작하는지

5. **모니터링 지표**
   - 401/403 발생률과 급증 알림
   - 만료 토큰 401 비율(정상적 갱신 흐름 여부)
   - 반복적 403(권한 오·남용 시도) 로깅

## 출력 계약

```
검증 결과: [통과 / 실패]
- 미인증 → 401: [정상 / 오류]
- 권한 부족 → 403: [정상 / 401로 잘못 처리]
- 역할별 접근: [매트릭스 전부 통과 / 실패]
- 만료 처리: [401 + 코드]
- 클라이언트 리다이렉트: [정상 / 문제]
```
