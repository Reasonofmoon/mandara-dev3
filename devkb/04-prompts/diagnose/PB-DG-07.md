# 인증/인가 401·403 원인 진단

접근 거부가 재현된 상태에서 토큰 누락, 토큰 만료, 잘못된 토큰, 권한(역할) 불일치 중 어느 것이 원인인지 401/403 경계를 기준으로 원인 후보를 정렬하고, 각 가설을 확정 또는 기각할 증거 수집 방법을 지시하는 진단 프롬프트입니다.

## 용도

401(인증 문제)과 403(인가 문제)을 먼저 갈라낸 뒤, 토큰 누락/만료/무효 또는 역할 부족 중 원인을 좁혀냅니다. 응답 상태 코드와 토큰 페이로드 대조로 검증합니다.

## 적용 시점

- "401 Unauthorized" 또는 "403 Forbidden"이 발생
- 로그인은 됐는데 특정 작업만 거부됨(403 의심)
- 한동안 잘 되다가 갑자기 거부됨(토큰 만료 의심)
- 401과 403이 구분 없이 처리되어 원인이 불명확함

## 필수 입력

- 요청 URL, 메서드, `Authorization` 헤더 유무
- 응답 상태 코드와 에러 바디(`code`, `message`)
- 토큰 페이로드(`role`/`roles`, `exp`)와 대상 라우트의 요구 권한
- 서버 인증/인가 미들웨어 코드(접근 가능하면)

## 프롬프트 템플릿

```
다음 인증/인가 접근 거부의 원인을 진단하라.

[요청]
URL/메서드: <...>
Authorization 헤더: <유무 및 값의 앞부분>

[응답]
상태 코드: <401 또는 403>
에러 바디: <error/code/message>

[토큰 페이로드 / 요구 권한]
<jwt.decode 결과, 라우트가 요구하는 role>

다음 순서로 진단하라:

1. 상태 코드로 1차 분기하라:
   - 401 → 인증 문제(토큰 없음/만료/무효)
   - 403 → 인가 문제(인증은 됐으나 권한 부족)

2. 원인 후보를 확률·검증비용 순으로 정렬하라:
   - (매우높음/저비용, 401) Authorization 헤더 자체가 없음
   - (높음/저비용, 401) 토큰 exp가 과거 → 만료
   - (높음/저비용, 401) 서명 검증 실패(시크릿 불일치)
   - (높음/중간, 403) 토큰 role이 라우트 요구 role에 포함되지 않음

3. 각 가설의 확정/기각 증거를 수집하라:
   - 토큰 누락: 헤더 존재 여부 확인. 없으면 401 → 누락이 원인
   - 만료: jwt.decode로 exp를 현재 시각과 비교.
     exp < now면 → 만료. 서버가 TokenExpiredError 코드를 반환하는지 대조
   - 무효 서명: 올바른 시크릿으로 재검증했을 때 통과하면 → 시크릿 불일치
   - 권한 부족: 403이면 토큰 role과 라우트 requiredRoles를 비교.
     포함 관계가 아니면 → 역할 불일치가 원인

4. 가장 확률 높고 검증 비용 낮은 가설부터 확정하고,
   확정된 원인과 증거를 명시하라. 기각된 가설도 근거와 함께 기록하라.
```

## 출력 계약

```
1차 분기: [401 또는 403]
확정 원인: [예: 토큰 exp가 지나 만료(401, code=TOKEN_EXPIRED)]
증거: [예: jwt.decode의 exp가 현재 시각보다 이전]
기각된 후보:
  - 토큰 누락: [기각 근거]
  - 권한 부족: [기각 근거]
다음 단계: [F-06 6단계 수정안 중 적용할 수정안 번호]
```
