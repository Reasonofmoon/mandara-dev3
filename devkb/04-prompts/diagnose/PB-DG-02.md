# React 무한 리렌더링 원인 진단

무한 리렌더링 증상이 재현된 상태에서 원인 후보를 확률 순으로 정렬하고, 각 가설을 확정 또는 기각할 증거 수집 방법을 지시하는 진단 프롬프트입니다.

## 용도

`setState` 루프, `useEffect` 의존성 오류, 객체 참조 불변성 위반, 상위 컴포넌트 프롭 중 어느 것이 원인인지 좁혀냅니다. React DevTools Profiler 근거로 각 가설을 검증합니다.

## 적용 시점

- "Maximum update depth exceeded" 경고가 콘솔에 반복 출력됨
- React DevTools Profiler에서 특정 컴포넌트의 렌더링 횟수가 폭증함
- CPU 사용률이 비정상적으로 높고 브라우저가 멈춤
- 증상은 재현되었으나 어느 훅/프롭이 원인인지 특정되지 않음

## 필수 입력

- 문제 컴포넌트의 소스 코드 (`useEffect`, `useState`, `useMemo`, `useCallback` 포함)
- 상위 컴포넌트에서 전달하는 props 목록
- React DevTools Profiler의 렌더링 커밋 스택 스냅샷
- 콘솔 경고 전문

## 프롬프트 템플릿

```
다음 React 무한 리렌더링 문제의 원인을 진단하라.

[컴포넌트 코드]
<문제 컴포넌트 전체 소스>

[상위 컴포넌트 props]
<전달되는 props와 그 생성 위치>

[Profiler 관찰]
<렌더링 커밋 횟수, 변경된 state/props>

다음 순서로 진단하라:

1. 원인 후보를 확률·검증비용 순으로 정렬하라:
   - (매우높음/저비용) useEffect 의존성 배열 누락 또는 렌더마다 변하는 값 포함
   - (높음/중간) 상위에서 매 렌더마다 새 객체/배열/함수를 prop으로 전달
   - (중간/중간) setState가 렌더 본문에서 조건 없이 호출됨
   - (낮음/고비용) 외부 라이브러리의 내부 상태 순환

2. 각 가설의 확정/기각 증거를 수집하라:
   - 의존성 누락: 해당 useEffect에 `console.log('effect run')`를 넣고,
     의존성 배열을 `[]`로 임시 고정했을 때 루프가 멈추면 → 의존성이 원인
   - 참조 불변성: 의심 prop을 `useRef`에 저장해 이전 렌더 값과
     `Object.is()`로 비교. 매 렌더마다 false면 → 새 참조 생성이 원인
   - 렌더 본문 setState: 렌더 함수 첫 줄에 로그를 찍어
     setState 호출 스택이 렌더 → setState → 렌더로 순환하는지 확인

3. 가장 확률 높고 검증 비용 낮은 가설부터 확정하고,
   확정된 원인과 그 증거를 명시하라. 기각된 가설도 근거와 함께 기록하라.
```

## 출력 계약

```
확정 원인: [예: useEffect 의존성 배열에 매 렌더마다 재생성되는 config 객체 포함]
증거: [예: config를 useMemo로 고정하자 effect run 로그가 1회로 감소]
기각된 후보:
```
