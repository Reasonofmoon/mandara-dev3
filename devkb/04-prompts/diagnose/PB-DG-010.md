# N+1 쿼리 성능 저하 원인 진단

쿼리 수 폭증이 재현된 상태에서 루프 내 쿼리, `include` 누락, 깊은 중첩 관계, GraphQL 리졸버 중 어느 것이 N+1을 유발하는지 원인 후보 순으로 정렬하고, 각 가설을 확정 또는 기각할 증거 수집 방법을 지시하는 진단 프롬프트입니다.

## 용도

Prisma 쿼리 로그를 근거로 반복문 내 개별 조회, 관계 미선택 로딩, 중첩 리졸버 중 N+1의 원인을 좁혀냅니다. 쿼리 카운트와 실행 로그 패턴 대조로 검증합니다.

## 적용 시점

- 작은 목록 조회에도 수십~수백 개의 쿼리가 발생
- 데이터 양이 늘수록 응답 시간이 선형 이상으로 악화
- DB CPU 사용률이 높고 동일 형태의 쿼리가 반복 출력됨
- 증상은 확인됐으나 어느 경로가 N+1을 유발하는지 미상

## 필수 입력

- 문제 경로의 데이터 조회 코드(Prisma 쿼리, 반복문/리졸버)
- Prisma 쿼리 로그(`log: ['query']`)의 실제 출력 샘플
- 조회 대상 모델의 관계 정의(1:N, M:N)
- 조회 건수와 그때 발생한 총 쿼리 수

## 프롬프트 템플릿

```
다음 N+1 쿼리 성능 문제의 원인을 진단하라.

[조회 코드]
<Prisma 쿼리, 반복문 또는 GraphQL 리졸버>

[쿼리 로그]
<log:['query']로 출력된 실제 쿼리 반복 패턴>

[모델 관계]
<schema.prisma의 관계 정의>

[관측]
조회 대상 건수: N
발생한 총 쿼리 수: <숫자>

다음 순서로 진단하라:

1. 먼저 쿼리 로그를 활성화하고 카운트를 확보하라:
   const prisma = new PrismaClient({ log: ['query'] });
   또는 prisma.$on('query', e => { count++; });
   총 쿼리가 부모 조회 1건 + 자식 조회 N건에 근접하면 N+1 강한 신호.

2. 원인 후보를 확률·검증비용 순으로 정렬하라:
   - (매우높음/저비용) 반복문(for/map/forEach) 안에서 관계를 개별 조회
   - (높음/저비용) findMany에 include/select 없이 이후 관계를 지연 조회
   - (중간/고비용) 여러 수준 중첩 관계를 각 수준마다 개별 조회
   - (낮음/고비용) GraphQL 리졸버가 필드마다 개별 쿼리(DataLoader 부재)

3. 각 가설의 확정/기각 증거를 수집하라:
   - 루프 내 쿼리: 조회 코드에서 반복문 내부의 await prisma... 호출을 찾는다.
     로그에 동일 형태 WHERE authorId = ? 쿼리가 N번 반복되면 → 루프 내 쿼리 확정
   - include 누락: 부모 findMany에 include가 없는지 확인.
     include: { 관계: true }를 추가했을 때 쿼리 수가 N+1에서 소수로
     떨어지면 → include 누락이 원인
   - 깊은 중첩: 로그가 1 + N + (N*M) 패턴을 보이면 → 다중 수준 중첩
   - GraphQL: 요청당 필드 리졸버마다 쿼리가 반복되면 → DataLoader 부재

4. 가장 확률 높고 검증 비용 낮은 가설부터 확정하고,
   확정된 원인과 증거를 명시하라. 기각된 가설도 근거와 함께 기록하라.
```

기각된 후보:
