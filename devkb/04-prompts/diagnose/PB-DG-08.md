# JWT 토큰 문제 원인 진단

JWT 관련 오류가 재현된 상태에서 시크릿 키 불일치, 만료 검증 오작동, 저장 위치 문제, 리프레시 토큰 결함 중 어느 것이 원인인지 원인 후보 순으로 정렬하고, 각 가설을 확정 또는 기각할 증거 수집 방법을 지시하는 진단 프롬프트입니다.

## 용도

"Invalid signature", 만료 미검증, 갱신 실패의 원인을 잘못된 시크릿, exp 설정/검증 오류, 저장 위치 취약성, 리프레시 로직 결함으로 좁혀냅니다. 토큰 디코드와 시크릿 대조로 검증합니다.

## 적용 시점

- "invalid signature" 또는 "jwt malformed" 오류 발생
- 만료됐어야 할 토큰이 계속 통과하거나, 정상 토큰이 너무 빨리 거부됨
- `/api/auth/refresh` 호출이 실패함
- 증상은 확인됐으나 생성/검증/저장/갱신 중 어디가 원인인지 미상

## 필수 입력

- 문제 토큰 문자열(민감 정보 마스킹 후 헤더·페이로드)
- 토큰 생성 코드(`jwt.sign` 옵션: `expiresIn`, `algorithm`, `issuer`, `audience`)
- 검증 코드(`jwt.verify` 옵션)와 사용 중인 시크릿 환경 변수 이름
- 리프레시 토큰 흐름과 저장 위치(쿠키/localStorage/메모리)

## 프롬프트 템플릿

```
다음 JWT 토큰 문제의 원인을 진단하라.

[토큰]
헤더/페이로드(디코드): <jwt.decode 결과, exp/iss/aud 포함>

[생성 코드]
<jwt.sign 옵션>

[검증 코드]
<jwt.verify 옵션, 시크릿 환경변수>

[갱신/저장]
<refresh 흐름, 저장 위치>

다음 순서로 진단하라:

1. 원인 후보를 확률·검증비용 순으로 정렬하라:
   - (매우높음/저비용) 생성과 검증이 서로 다른 시크릿 키 사용
   - (높음/저비용) 생성 시 expiresIn 누락 → 만료 없는 토큰
   - (높음/중간) verify 시 issuer/audience 옵션 불일치로 거부
   - (높음/중간) 리프레시 토큰이 액세스 시크릿으로 서명/검증되어 충돌

2. 각 가설의 확정/기각 증거를 수집하라:
   - 시크릿 불일치: 생성·검증 경로에서 참조하는 환경변수가 동일한지 확인.
     jwt.verify(token, 생성시_시크릿)이 통과하면 → 검증 측 시크릿이 원인
   - exp 누락: jwt.decode(token)에 exp 필드가 있는지 확인.
     없으면 → expiresIn 미설정
   - iss/aud 불일치: verify 옵션의 issuer/audience와 페이로드의 iss/aud 대조.
     옵션을 제거했을 때 통과하면 → iss/aud 불일치가 원인
   - 리프레시 시크릿: refresh 검증이 REFRESH_SECRET을 쓰는지,
     생성도 같은 시크릿인지 확인. 다르면 → 리프레시 시크릿 충돌

3. 가장 확률 높고 검증 비용 낮은 가설부터 확정하고,
   확정된 원인과 증거를 명시하라. 기각된 가설도 근거와 함께 기록하라.
```

증거: [예: 생성 시크릿으로 jwt.verify 시 통과, 검증 코드 시크릿으로는 invalid signature]
