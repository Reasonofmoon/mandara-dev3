# API 멱등성 결여 원인 진단

같은 요청이 여러 결과를 만드는 증상이 재현된 상태에서 Idempotency-Key 부재, 중복 검사 누락, DB 제약 부족 중 어느 지점이 중복을 허용하는지 원인 후보 순으로 정렬하고, 각 가설을 확정 또는 기각할 증거 수집 방법을 지시하는 진단 프롬프트입니다.

## 용도

동일 요청 반복 시 리소스가 중복 생성되는 원인을 Idempotency-Key 미구현, 서버 중복 검사 부재, DB 유니크 제약 부족, 재시도 로직 부재로 좁혀냅니다. 반복 요청 결과의 ID 비교로 검증합니다.

## 적용 시점

- 같은 결제/주문 요청을 여러 번 보내면 매번 새 리소스가 생김
- 네트워크 오류 후 재시도가 중복 처리를 유발함
- 응답에 `Idempotency-Replay` 헤더가 없음
- 증상은 확인됐으나 서버/클라이언트/DB 중 어디가 원인인지 미상

## 필수 입력

- 대상 엔드포인트의 메서드와 핸들러 코드
- 클라이언트 요청의 헤더(특히 `Idempotency-Key` 유무)와 재시도 로직
- 리소스 테이블 스키마(유니크 제약 유무)
- 반복 요청 시 생성된 리소스 ID 목록

## 프롬프트 템플릿

```
다음 API 멱등성 문제의 원인을 진단하라.

[엔드포인트 핸들러]
<메서드와 create 로직>

[클라이언트 요청]
<헤더, 재시도 로직>

[테이블 스키마]
<유니크 제약 관련 컬럼>

다음 순서로 진단하라:

1. 원인 후보를 확률·검증비용 순으로 정렬하라:
   - (매우높음/중간) 클라이언트가 Idempotency-Key를 보내지 않음
   - (높음/중간) 서버가 Key를 받아도 이전 요청 존재 여부를 검사하지 않음
   - (중간/중간) DB에 idempotency_key 유니크 제약이 없어 중복 삽입 허용
   - (높음/저비용) 재시도 로직이 매번 새 Key를 생성

2. 각 가설의 확정/기각 증거를 수집하라:
   - Key 미전송: 같은 Idempotency-Key로 3회 요청해 결과 ID 비교:
     for i in {1..3}; do curl -X POST <url> \
       -H "Idempotency-Key: fixed-123" -d '<body>'; done
     ID가 매번 다르면 → 서버가 Key를 활용하지 않는 것
   - 서버 중복 검사: 핸들러에서 findOne({idempotencyKey}) 또는
     Redis 조회가 있는지 코드로 확인. 없으면 → 중복 검사 부재
   - DB 제약: 스키마에서 idempotency_key가 UNIQUE인지 확인.
     아니면 동시 요청 시 중복 삽입 가능 → DB 제약 부족
   - 재시도 Key: 클라이언트 재시도 코드가 루프 밖에서 Key를 한 번만
     생성하는지 확인. 루프 안에서 생성하면 → 재시도마다 새 Key가 원인

3. 가장 확률 높고 검증 비용 낮은 가설부터 확정하고,
   확정된 원인과 증거를 명시하라. 기각된 가설도 근거와 함께 기록하라.
```
