# N+1 쿼리 성능 문제 파악

DB 쿼리가 폭증하는 성능 저하가 의심될 때, 어떤 반복 지점에서 관계 데이터가 개별 조회되는지 짚어내도록 유도하는 질문 세트입니다.

## 용도

"느려요"를 넘어, 루프 내 쿼리·관계 미선택 로딩·중첩 관계·GraphQL resolver 중 어디서 N+1이 발생하는지 좁힙니다.

## 적용 시점

- 작은 목록 조회에도 수십~수백 개의 쿼리가 발생함
- 데이터 양이 늘수록 응답 시간이 급격히 나빠짐
- DB CPU 사용률이 비정상적으로 높음
- 페이지/엔드포인트 로드가 오래 걸림

## 필수 입력

- 문제 조회 코드 (findMany + 루프, include/select 사용 여부)
- Prisma 쿼리 로그 또는 발생 쿼리 개수
- 조회하는 관계(1:N / M:N)와 중첩 깊이

## 프롬프트 템플릿

아래 질문에 답해 N+1 발생 지점을 좁혀주세요.

1. **쿼리 수 측정**
   - 쿼리 로깅(`log: ['query']` 또는 `$on('query')`)을 켜고 한 번의 요청에 쿼리가 몇 개 나가나요?
   - 조회하는 상위 레코드 수(N)에 비례해 쿼리 수가 늘어나나요? (1 + N 패턴)

2. **반복 지점 확인**
   - `for`/`map`/`forEach` 루프 안에서 `findMany`, `findUnique`, `count` 같은 쿼리를 호출하나요?
   - 루프 안에 또 루프가 있어 `1 + N + N*M`처럼 폭증하나요?

3. **관계 로딩 방식**
   - 관계 데이터를 `include`나 `select`로 한 번에 로드하나요, 아니면 개별 조회하나요?
   - 필요 없는 필드까지 다 가져오나요? 필요한 필드만 `select`하나요?

4. **중첩·집계**
   - 여러 수준(user→posts→comments)의 관계를 조회하나요? 어느 수준에서 개별 쿼리가 터지나요?
   - 개수(count) 같은 집계를 레코드마다 따로 구하나요? (groupBy로 한 번에 가능한 것)

5. **GraphQL·범위**
   - GraphQL resolver에서 필드마다 쿼리가 나가나요? DataLoader로 배치하고 있나요?
   - 특정 엔드포인트/화면에서만 느린가요, 전반적인가요?

## 출력 계약

N+1 쿼리 문제 정의:
- 증상: [요청당 쿼리 수, N 비례 증가 여부]
- 발생 지점: [루프 내 쿼리 / 중첩 루프 / resolver]
- 관계 로딩: [include/select 사용 여부]
- 중첩·집계: [중첩 깊이, 레코드별 count 여부]
- 범위: [특정 엔드포인트 / 전반]

## 셀프 체크리스트

- [ ] 쿼리 로깅으로 요청당 쿼리 수를 측정했는가?
- [ ] 쿼리 수가 N에 비례해 증가하는지 확인했는가?
- [ ] 루프(중첩 포함) 내 쿼리 호출을 특정했는가?
- [ ] include/select 사용 여부를 확인했는가?
- [ ] GraphQL resolver의 DataLoader 사용 여부를 확인했는가?
