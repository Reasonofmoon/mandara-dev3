# API 멱등성 문제 파악

같은 요청이 여러 번 처리되어 부작용이 생길 때, 어떤 엔드포인트가 어떤 재시도 경로에서 중복을 만드는지 짚어내도록 유도하는 질문 세트입니다.

## 용도

"결제가 두 번 됐어요"를 넘어, Idempotency-Key 유무·중복 검사·재시도 로직·DB 제약 중 무엇이 빠졌는지 좁힙니다.

## 적용 시점

- 같은 요청을 여러 번 보내면 리소스가 중복 생성됨(주문·결제 등)
- 네트워크 오류 후 재시도 시 중복 처리됨
- 재시도 로직이 부작용을 일으킴
- 분산 환경에서 부분 실패 후 재실행으로 불일치 발생

## 필수 입력

- 문제 엔드포인트의 메서드와 경로 (예: POST /api/orders)
- 클라이언트 재시도 로직 코드 (있다면)
- 요청이 중복을 만드는 재현 시나리오

## 프롬프트 템플릿

아래 질문에 답해 중복 발생 지점을 좁혀주세요.

1. **증상 확인**
   - 어떤 리소스가 중복 생성되나요? (주문/결제/가입 등) 한 번의 사용자 행동으로 몇 개가 생기나요?
   - 재시도로 인한 것인가요, 사용자의 더블클릭 때문인가요, 아니면 서버 재처리 때문인가요?

2. **엔드포인트 특성**
   - 문제 엔드포인트의 메서드가 POST인가요? PUT/PATCH/DELETE라면 의도상 멱등이어야 하나요?
   - 서버가 요청마다 무조건 `create`를 호출하나요, 아니면 기존 리소스 존재 여부를 먼저 확인하나요?

3. **Idempotency-Key 유무**
   - 클라이언트가 `Idempotency-Key` 헤더를 보내나요? 재시도 시 같은 키를 재사용하나요, 매번 새로 만드나요?
   - 서버가 그 키로 이전 응답을 조회·재사용하나요? (Redis/DB 저장 여부)

4. **재시도·중복 방지 로직**
   - 클라이언트에 재시도(지수 백오프 등) 로직이 있나요? 실패 판정 기준은 무엇인가요?
   - DB에 유니크 제약(예: idempotency_key UNIQUE)이 걸려 있나요?

5. **분산·범위 확인**
   - 여러 서비스에 걸친 트랜잭션인가요? 부분 실패 시 롤백/보상 처리가 있나요?
   - 특정 부하 상황(동시 요청)에서만 발생하나요, 항상 재현되나요?

## 출력 계약

멱등성 문제 정의:
- 증상: [중복 리소스 종류와 개수]
- 트리거: [재시도 / 더블클릭 / 서버 재처리]
- 엔드포인트: [메서드·경로, 존재 확인 로직 유무]
- Idempotency-Key: [전송 여부, 키 재사용 여부, 서버 조회 여부]
- DB 제약: [유니크 제약 유무]
- 범위: [단일 서비스 / 분산 트랜잭션, 재현 조건]

## 셀프 체크리스트

- [ ] 중복 생성되는 리소스와 개수를 확인했는가?
- [ ] 중복 트리거(재시도/더블클릭/재처리)를 특정했는가?
- [ ] Idempotency-Key 전송·재사용·서버 조회 여부를 확인했는가?
- [ ] DB 유니크 제약 유무를 확인했는가?
- [ ] 분산 트랜잭션·동시성 조건을 파악했는가?
