# React 무한 리렌더링 문제 파악

무한 리렌더링이 의심될 때, 원인 코드를 지목하기 전에 어떤 렌더링 루프가 돌고 있는지 정확히 관찰·기술하도록 유도하는 질문 세트입니다.

## 용도

개발자가 "화면이 멈췄다", "느리다"는 막연한 표현 대신, 어떤 state/effect가 렌더링을 반복 트리거하는지 구체적 증거로 좁히도록 돕습니다.

## 적용 시점

- 콘솔에 "Warning: Maximum update depth exceeded"가 반복 출력됨
- CPU 사용률이 비정상적으로 높고 탭이 멈춤
- Network 탭에 동일 API 요청이 끊임없이 반복됨
- React DevTools Profiler에서 렌더링 횟수가 폭증함

## 필수 입력

- 문제가 되는 컴포넌트 코드 (특히 useEffect / setState 부분)
- 콘솔 경고 메시지 전문
- 렌더링을 유발한다고 의심되는 state 또는 prop

## 프롬프트 템플릿

아래 질문에 순서대로 답해주세요. 답을 채우면 무한 루프의 출처가 좁혀집니다.

1. **증상 관찰**
   - 콘솔에 "Maximum update depth exceeded" 경고가 뜨나요? 전문을 붙여주세요.
   - React DevTools Profiler에서 문제 컴포넌트의 렌더링 횟수가 초당 몇 회인가요?
   - Network 탭에서 같은 API 요청이 무한 반복되나요?

2. **useEffect 점검**
   - 루프가 의심되는 useEffect의 의존성 배열은 어떻게 되어 있나요? (배열 없음 / 빈 배열 / 특정 값)
   - 그 useEffect 안에서 setState를 호출하나요? 호출한다면 어떤 state인가요?
   - 의존성 배열에 객체·배열·함수를 직접 넣고 있나요? (예: `[{ id: 1 }]`, `[handler]`)

3. **상태 업데이트 위치**
   - 렌더링 본문(return 이전)에서 직접 setState를 호출하는 곳이 있나요?
   - setState 시 이전 상태를 참조하나요? (`setCount(count + 1)` vs `setCount(prev => prev + 1)`)

4. **상위 컴포넌트 / prop**
   - 부모가 매 렌더링마다 새 함수·객체를 자식에게 prop으로 넘기나요?
   - 문제 컴포넌트를 부모에서 제거하면 루프가 사라지나요?

5. **범위 확인**
   - 특정 컴포넌트 하나만 문제인가요, 아니면 트리 전체가 반복 렌더되나요?
   - 최근 어떤 코드 변경 직후부터 발생했나요?

## 출력 계약

무한 리렌더링 문제 정의:
- 증상: [경고 메시지 / 렌더링 횟수 / 반복 요청 여부]
- 트리거 state/effect: [의심 대상과 의존성 배열 상태]
- setState 위치: [렌더 본문 / effect 내부 / 이벤트 핸들러]
- prop 참조 불안정성: [부모가 넘기는 새 객체·함수 여부]
- 재현 범위: [단일 컴포넌트 / 서브트리]

## 셀프 체크리스트

- [ ] 콘솔 경고 전문을 확보했는가?
- [ ] 의심 useEffect의 의존성 배열 상태를 확인했는가?
- [ ] setState 호출 위치를 특정했는가?
- [ ] 부모가 전달하는 prop의 참조 안정성을 확인했는가?
- [ ] 렌더링 횟수를 Profiler로 측정했는가?
